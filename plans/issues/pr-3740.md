## Why this matters

The dependabot bump **provably breaks the build**: CI's `setup` job fails with


"turbo 1→2" as its own opt-in migration. So: close the bot PR, and (gated)
open a proper migration PR.

`gh pr close 3740 --comment "Closing: turbo 2 requires a config migration dependabot can't author (turbo.json pipeline→tasks, --no-cache flag removal) — this bump alone breaks the build (CI setup job fails on turbo.json:2). Tracked as a proper migration in plans/issues/pr-3740.md."`

### Step 2 (GATE): replacement migration PR

If the `plans/issues/README.md` row is not marked APPROVED, mark it
"close done; migration awaiting approval" and stop. If approved, on a new
branch `migrate/turbo-2`:

1. `turbo.json`: rename the top-level `pipeline` key to `tasks` (content
   unchanged).
2. `package.json`: `"turbo": "1.x"` → `"2.x"`; in `test-ci`, replace
   `--no-cache` with `--cache=local:r,remote:r`.
3. `yarn install` to regenerate the lockfile (one clean foreground install,
   per CLAUDE.md).
4. Read the turbo 2 upgrade notes for env-mode changes: turbo 2 defaults to
   strict env mode. Check `turbo.json` for `globalEnv`/`env` declarations; if
   tasks rely on undeclared env vars (e.g. `TEST_PORT` used by dev servers,
   CI tokens), declare them. Grep CI config (`.circleci/config.yml`) for env
   vars consumed by turbo-run scripts.
5. Local verification: `yarn build` (root) exit 0; `yarn test` passes (modulo
   known SSR TextEncoder / use-velocity failures); `npx turbo run build --dry`
   exits 0 and lists the same task graph as before.
6. Open the PR (`gh pr create`), wait for ALL five CI jobs green (setup, test,
   test-html, test-react, test-react-19) — the setup job is the one that
   failed before, but the dev-server-dependent E2E jobs are where env-mode
   regressions would surface. Merge on green.

**Verify**: each numbered sub-step's command output as stated.
