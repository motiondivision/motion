## Why this matters

The PR's own body concedes the case: "I could not reproduce the reported
flicker… Rather than landing speculative code changes for a bug I cannot
trigger, this PR locks in the desired behavior." That is precisely the pattern
this repo's recorded policy rejects (memory: "no repro → no fix, no
speculative coverage" — if the test can't fail on the bug, close as needs-repro
rather than landing happy-path tests). The test samples computed opacity at
2.5s/5s/7.5s wall-clock with ±0.1 tolerance — it would only catch the reported
~50% *flicker* if the flicker coincided with a sample instant; it guards
against gross easing regressions, which existing animation tests already
cover. Wall-clock `cy.wait()` sampling also adds flake surface to a CI suite
already plagued by it (Greptile flagged exactly this, 4/5).
